# Wearable-Verified Loyalty — Стратегическая концепция

> Добавлено: 2026-06-16
> Статус: концепт для следующей итерации продукта

---

## Суть идеи

Текущая система начисляет поинты за чекин — это легко гамить и не отражает
реального усилия. Wearable-верификация закрывает эту дыру:

```
Чекин в зале (QR / геофенс)
    +
Данные трекера за следующие 60-90 минут
(активные минуты, пульс, калории)
    =
Подтверждённая тренировка → поинты на основе реального усилия
```

**Ключевой дифференциатор:** единственная gym loyalty система которая
награждает за работу, а не за присутствие. Нельзя накрутить сидя в раздевалке.

---

## Уровни интенсивности (маппинг на существующий UI)

Heatmap levels 0–4 уже есть в продукте. Маппинг на wearable данные:

| Уровень | Значение | Данные трекера | Поинты |
|---|---|---|---|
| 0 | Не приходил | — | 0 |
| 1 | Лёгкая активность | 20-30 мин, низкий пульс | 50 |
| 2 | Нормальная тренировка | 40-60 мин, умеренный пульс | 100 |
| 3 | Хорошая тренировка | 60+ мин или высокий пульс | 150 |
| 4 | Интенсивная | 45+ мин, зона 4-5 пульса | 200 |

Базовые 50 очков за чекин остаются → никто не исключён. Трекер = усилитель.

## Поддерживаемые источники данных

Все через стандартные OAuth API, не требуют специального железа:

| Источник | Платформа | Что даёт |
|---|---|---|
| **Apple HealthKit** | iOS | Шаги, активные минуты, HR, калории, тренировки |
| **Google Health Connect** | Android | То же (Health Connect SDK v1.1+) |
| **Fitbit Web API** | iOS + Android | Подробные HR zones, интенсивность |
| **Garmin Connect** | iOS + Android | Профессиональные данные, популярен у бегунов/велосипедистов |
| **Strava API** | iOS + Android | GPS треки, активности — для бега и велоспорта |

**MVP — достаточно двух:** Apple HealthKit + Google Health Connect покрывают
~90% смартфонов без дополнительного железа. iPhone считает активные минуты
автоматически без Apple Watch.

## Верификация без фрода

Комбинация двух сигналов исключает накрутку:

1. **Геолокация:** чекин через QR на ресепшн или геофенс (радиус 250м)
   подтверждает что человек физически в зале
2. **Биосигнал:** Active Minutes или HR в пульс-зоне 3+ в течение 60-90 мин
   после чекина подтверждает что тренировка реально была

Оба сигнала обязательны → невозможно накрутить без физического присутствия
и физической нагрузки.

## Первый вертикаль: почему залы и фитнес-студии

| Критерий | Оценка |
|---|---|
| Уже построено (B2B инструментарий) | ✅ |
| Контролируемая среда (один адрес = геофенс) | ✅ |
| Платящий клиент уже определён (владелец зала) | ✅ |
| Wearable-проникновение у ЦА | ✅ ~60-70% у регулярных посетителей |
| Progressive rollout (чекин → чекин+трекер) | ✅ |
| Eversports интеграция в роадмапе | ✅ |
| Krankenkassen-синергия (те же данные визитов) | ✅ |

**Лучший первый тип студий:** йога и пилатес (100-250 членов)
- Классовый формат = чёткие временны́е окна для верификации
- Члены более склонны к health tracking
- Eversports — основной софт (наша аудитория)
- Нет тяжёлого железа (турникеты и т.п.) = простой чекин

Та же механика, разные методы верификации активности:

| Активность | Верификация | Бизнес-получатель наград |
|---|---|---|
| **Бег** | GPS маршрут (Strava/Garmin), дистанция ≥3км | Кафе на маршрутах, бегшопы |

| **Хайкинг** | GPS + набор высоты ≥200м | Горные кафе, аутдор-магазины |
| **Плавание** | Apple Watch / Garmin swim | Бассейны, спортмагазины |

## DSGVO-позиция

HR и биометрия → Art. 9 GDPR (специальная категория данных).
**Чистое решение:** не хранить сырые биометрические данные. Хранить только
бинарный результат верификации ("workout confirmed: yes/no") и уровень 1-4.
Данные обрабатываются на устройстве или в памяти, не персистируются.
Это "zero-knowledge biometrics" позиция — сильный маркетинговый аргумент.

## Архитектура доставки приложения (решено 2026-06-17)

### Ключевое ограничение

Чистая PWA **не может** читать `Apple HealthKit` и `Google Health Connect` —
это нативные on-device API без веб-интерфейса. Трекеры делятся на два типа:

| Тип | Источники | Доступ из PWA |
|---|---|---|
| **Облачные (cloud OAuth)** | Fitbit, Garmin, Strava, Oura, Withings, Polar | ✅ да — backend тянет из облака вендора, телефон не нужен; webhooks решают и проблему фоновой синхронизации после чекина |
| **On-device** | Apple Watch / iPhone active minutes, Health Connect | ❌ нет — только через нативную обёртку |

**Apple Watch — критичный кейс:** его данные живут только в HealthKit на iPhone,
облачного "Apple Health Web API" не существует. А первый вертикаль (йога/пилатес)
— iPhone-тяжёлый, поэтому Apple Watch обязателен.

### Решение: Capacitor + агрегатор Terra

**Путь B — обернуть PWA в Capacitor**, опубликовать в App Store / Play Store.
Тот же веб-код (95%+ переиспользуется) работает в нативном шелле с мостами к
HealthKit и Health Connect через плагины. Получаем всё: Apple Watch + iPhone
active minutes + Health Connect + облачные OAuth.

- **Capacitor** (от Ionic) оборачивает существующий веб-UI в WebView + нативные
  плагины. В отличие от Expo (React Native — нужно переписывать UI нативно),
  Capacitor сохраняет наш веб-код как есть → меньше переписывания. Член-апп
  собираем как client SPA (не Next SSR), Capacitor указывает на этот бандл.
- **Terra** (tryterra.co) — агрегатор: одна интеграция вместо 30+. Для Apple/Google
  даёт нативный SDK (вставляется в Capacitor), для облачных — серверный OAuth +
  webhooks. Нормализует HR-зоны / active minutes в единую схему под уровни 1-4.
  Альтернативы: Rook, Vital, Spike. Платно (per-user), но экономит недели на MVP.

Путь A (чистая PWA, только облачные трекеры, без Apple Watch) держим как
fallback-формулировку, не как основную.

### White-label: как кастомизация работает через дашборд

Это **runtime data-driven theming**, а не отдельный бинарник на каждый зал —
стандартный SaaS-подход, технически полностью реален:

```
Админ в дашборде: "Branding" → logo upload, accent color, app name, reward catalog
    → сохраняется в tenant/branding запись в БД
Член-апп при запуске: GET /api/branding?gym=… → { name, logoUrl, colorPrimary }
    → выставляет CSS-переменные → рендерит под бренд зала
```

Ни ребилда, ни ре-сабмита на каждого клиента. Уровни:

- **Starter / Growth:** один мульти-тенант app в сторах, брендинг подтягивается
  по логину/коду зала в рантайме. В PWA-режиме на поддомене зала
  (`gym.fitloyalty.app`) ещё и динамический `manifest.webmanifest` → иконка и
  имя на home-screen уже бренда зала.
- **Pro (pf3 "Custom app domain & branding"):** выделенный бинарник зала со
  своей иконкой/листингом в сторе. Дорого (build-pipeline + риск Apple Guideline
  4.3 о дубль-приложениях → сабмит под аккаунтом клиента) → апсейл Pro-тира.

Theming и доступ к health — ортогональны: способ брендинга никак не влияет на
интеграцию трекеров.
