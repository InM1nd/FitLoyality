## Принцип: гейты, не чек-лист

Не внедрять всё сразу — каждый уровень привязан к конкретному триггеру в
развитии проекта. Внедрение раньше триггера = преждевременная инфраструктура
под несуществующую команду/бэкенд.

## Уровень 0 — сделать сейчас (защищает демо, на который уже идёт трафик)

**Триггер:** уже активен (холодный аутрич по email/Instagram идёт, см. история
чата — расширение вертикалей EMS/Boxen/Dance).

1. **Минимальный CI** — GitHub Actions, запуск на push/PR:
   - `npm run build`
   - `npm run lint`
   - `npm run typecheck` (скрипт нужно добавить: `tsc --noEmit`)
   Цель — не дать сломанной сборке доехать до прода без единого автоматического
   чека (сейчас ничего этому не мешает).

2. **1-2 unit-теста на `lib/churn-check.ts`.** Не про покрытие в принципе —
   это единственный кусок реальной бизнес-логики, с которым незнакомый
   прошпект взаимодействует без вас рядом (загружает свой CSV на лендинге).
   Регресс здесь молча ломает главный lead-magnet посреди аутрич-кампании.
   Нужен test-runner (Vitest — самый лёгкий вход для Next.js/TS, не требует
   доп. конфигурации типа Jest+Babel).

3. **Подтвердить стабильность Vercel-деплоя** — preview на PR, прод деплоится
   с правильной ветки (не `main` посреди незавершённой работы). Проверить
   через Vercel dashboard/CLI, не через репозиторий (там ничего не найдено).

**Оценка усилия:** 1-2 часа. **Не делать больше этого на Уровне 0** — соблазн
добавить полноценный test-suite здесь — уже Уровень 1.

**Триггер:** НЕ "когда-нибудь", а конкретно — когда первая студия (см.
`MARKET_RESEARCH.md` §13 GTM, design-партнёры Вены) даёт реальные CSV/данные
вместо mock, или когда начинается работа над реальным бэкендом.

- Полноценный test-фреймворк: Vitest для unit, Playwright (уже доступен как
  skill в этом окружении) для e2e smoke 2-3 ключевых демо-флоу (Churn-Check,
  guided tour, Setup Wizard).
- `middleware.ts` / auth — реализовать то, что осознанно отложено в
  `docs/AUDIT.md` №6 (плейсхолдер с TODO уже предлагался там).
- Разделение env: staging vs prod, раз появятся реальные API-ключи вместо
  мок-данных из `lib/mock-data.ts`.
- Error monitoring (Sentry или аналог) — иначе о поломке демо/продукта узнаете
  только когда прошпект/партнёр не ответит, а не из алерта.

## Уровень 2 — гейт: PMF + рост команды

**Триггер:** появление второго разработчика/агента, работающего в репо
параллельно, или первые платящие клиенты сверх design-партнёров.

- Pre-commit hooks (husky + lint-staged).
- Полноценные CI-гейты (blocking checks на PR, обязательный review).
- Тестовое покрытие по явным правилам (не "чем больше, тем лучше" — целевые
  зоны: биллинг-логика, data layer `lib/data.ts`, white-label theming).
- Staged deployment pipeline (staging → prod с промежуточным ревью).

**Не внедрять до триггера** — на соло-стадии это чистый overhead без пользы.

## Как этим пользоваться (для агентов)

- Перед любой задачей, которая выглядит как "добавить тесты"/"настроить CI" —
  сначала проверить, какой Уровень сейчас релевантен (см. триггеры выше), а не
  внедрять максимум по умолчанию.
- Уровень 0 можно/нужно закрывать независимо от бизнес-стадии — это не про
  масштаб, а про базовую защиту того, что уже публично доступно.
- Уровни 1/2 — не начинать без явного триггера от пользователя ("подписали
  первого партнёра", "беру второго разработчика" и т.п.), даже если "по
  чек-листу давно пора".
- Статус пунктов не отслеживается отдельной таблицей (в отличие от
  `docs/AUDIT.md`) — при выполнении Уровня 0/1/2 обновлять этот файл inline
  (дописывать "✅ сделано <дата>" к конкретному пункту), не создавать новый
  документ.
